# 網路件核卡額度商業邏輯

## 概述

原持卡人加辦信用卡的核卡額度處理邏輯，主要涉及信用額度與卡片額度的管理機制。

## 核卡規則

### 原持卡人加辦信用卡

- **財力證明**：原持卡人加辦信用卡無須提供財力證明
- **核卡方式**：
  - 系統自動核卡：直接以卡人 PCIC 的額度核卡
  - 人工核卡：同樣以卡人 PCIC 的額度核卡
- **額度設定**：建議調整為使用「信用額度」作為核卡依據

## 額度類型說明

### 1. 卡片額度

- **定義**：每張信用卡個別的最高刷卡上限
- **特性**：針對單張卡片的額度限制

### 2. 信用額度

- **定義**：以正卡持有人為基礎的歸戶總額度
- **特性**：所有信用卡共享的最高信用額度
- **系統管理**：主機主要管理並檢核信用額度 (CR_CRLIMIT)
- **授權參考**：授權時會參考歸戶的信用額度

## 額度調整機制

### 系統自動調整規則

| 情況                      | 處理方式                               |
| ------------------------- | -------------------------------------- |
| 新核卡額度 < 現有信用額度 | 系統不修改目前的信用額度               |
| 新核卡額度 > 現有信用額度 | 系統更新信用額度，將其提高至新核卡額度 |

### 額度關係

- 卡片額度與信用額度可能相同，但並不總是相同
- 信用額度為所有卡片的共享上限

## 授權檢核機制

### 刷卡授權順序

1. **第一層檢核**：檢查卡片信用額度可用餘額
   - 確認刷卡的該張卡片是否有足夠可用餘額
2. **第二層檢核**：檢查歸戶信用額度可用餘額
   - 若單張卡片餘額不足，檢查所有信用卡共享的信用額度可用餘額

### 授權結果

- **通過條件**：消費金額 ≤ 可用餘額
- **拒絕條件**：餘額不足

## 系統實作考量

### 資料來源問題

- **現況**：原額度目前從 CUST_INFO 抓取
- **需求**：需要能夠抓取原持卡人之前的核卡金額
- **建議**：改用信用額度作為核卡依據，因為主機主要管理信用額度

### 技術建議

- 核卡時優先參考信用額度 (CR_CRLIMIT)
- 確保系統能正確讀取並更新歸戶信用額度
- 維持授權檢核的雙層機制
